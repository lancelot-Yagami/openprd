# 创业验证透镜

当 L2 需求、脑暴模式或 0 到 1 方向讨论还停留在“做什么功能”时，用这组问题把它收回到更真实的验证闭环。

## 六个关键问题

1. 第一批最容易触达的社区、渠道或人群是谁？
2. 他们现在主要靠什么替代方案、流程或人工方式在解决？
3. 如果先不做完整产品，怎么靠手工服务、半自动流程或轻量交付先把价值跑出来？
4. 什么真实承诺最能证明这不是口头兴趣？
5. 最低成本先验证哪一步最关键？
6. 在验证阶段，怎样控制成本、时间和交付方式，确保这件事先活下来？

## 使用方式

- 不要机械逐条盘问；优先把它融进 requirement 摘要、脑暴卡片和追问顺序里。
- 如果用户已经明确提供其中几项，就直接吸收，不要重复确认。
- 对内部工具、团队流程或 Agent 项目，也照样适用。
  这里的“社区 / 种子用户”可以是维护者、运营、交付、内部试点团队或最先受影响的一线角色。
- “承诺信号”不等于一定要收费。
  也可以是愿意预约、提交真实素材、导入数据、拉同事参与、接受试点、给出明确上线窗口等更强承诺。
- “default alive”不是财务口号，而是提醒 Agent 先问清楚：
  这件事在验证阶段怎样不把成本、复杂度和维护负担一下子做爆。
