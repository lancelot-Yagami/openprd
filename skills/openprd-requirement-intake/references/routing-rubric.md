# Requirement Routing Rubric

## 判断维度

按下面四个维度判断，不按固定关键词判断。用户审查时优先展示“需求类型：直接处理（L0）”这类合并标签；只有内部调度和排障真的受益时，才额外附上“内部路由码”。

| 用户可见需求类型 | 内部路由码 | 默认路径 |
|---|---|---|
| 直接处理 | L0 | 直接处理，完成后说明验证 |
| 现有功能优化 | L1 | 先给 mini-plan，再实现和验证 |
| 新功能/新流程方案 | L2 | 先澄清并进入 PRD/review/change/tasks |

| 维度 | L0 | L1 | L2 |
|---|---|---|---|
| 影响面 | 单点文件、单个 UI 文案、局部 bug | 多文件或多状态，但边界清楚 | 跨模块、跨系统、跨角色、跨端或跨任务 |
| 未知数 | 几乎没有 | 有少量实现选择 | 目标、用户、范围、验收或风险未定 |
| 决策成本 | 不需要产品决策 | 需要简短方案选择 | 需要先评审意图和范围 |
| 验证成本 | 单元/局部验证 | 组合验证或 E2E | 需要分阶段任务、回归、业务/成本/权限验证 |

## UI 影响面补充判断

界面、页面、视觉、样式或前端体验需求先按 L0/L1/L2 分流，再判断是否需要视觉方案评审。

- 小 UI 调整：按钮文案、颜色、间距、字号、单点样式、局部 bug，通常仍按 L0 或 L1。
- 大界面改动：信息架构、核心布局、主视觉、关键路径、组件层级/密度明显变化，或用户需要先选设计方向。
- 大界面改动不直接实现；在 mini-plan、PRD 定稿或实现开工前，已有界面时先用 Computer Use 截当前产品内功能图，冷启动没有现有界面时基于已确认 PRD、用户群体、第一版切片和视觉目标生成设计 brief；再用 Image 2 生成至少 3 个方向，横向拼接带 1/2/3 序号的大图给用户确认。

## L0：直接处理

适合直接处理：

- 错别字、空格、标点、按钮文案小改
- 明确 bugfix，根因和验收都很窄
- 简单样式调整，不改变信息架构或用户流程
- 单个配置值或展示值修正

输出：需求类型为“直接处理”，直接做，完成后说明验证。不要生成 PRD。

## L1：现有功能优化

适合 mini-plan：

- 目标明确，但需要改多个文件
- 用户可见行为变化，但不引入新角色、新流程或新依赖
- 一个现有功能或现有流程里的局部体验优化
- 一个明确 bug 的系统化修复，需要补测试和文档

输出：需求类型为“现有功能优化”，先给 3-5 行 mini-plan，包含范围内、范围外、验证方式。若属于大界面改动，先补视觉方案评审；用户确认方向后才进入大 UI 实现。

## L2：新功能/新流程方案

必须进入 PRD：

- 新产品、新模块、新入口、新工作流或新信息架构
- 账号、权限、支付、订阅、计费、审批、审计、团队协作
- AI Agent、模型调用、第三方 API、云服务、成本或滥用风险
- 数据迁移、资源链路切换、发布链路、跨端/跨系统集成
- 用户说“继续任务”但补充了会改变范围的新目标
- 影响面和验收不清，Agent 需要猜产品决策

输出：需求类型为“新功能/新流程方案”，先澄清并进入 PRD/review/change/tasks。

## 续接规则

当用户提供会话 ID、work unit、task handle 或说“继续这个任务”：

1. 先用 `openprd run . --context --message <用户原话>` 解析显式目标。
2. 判断用户这次新增的话是否仍在原 PRD 范围内。
3. 如果只是继续已拆任务，按 tasks 继续。
4. 如果新增目标改变范围、验收、角色、依赖或风险，先更新 PRD/change/tasks。
5. 不要让当前 active change 抢走用户明确给出的会话或任务目标。

## 反例

- “把按钮文字从 A 改成 B”不是 L2。
- “修复支付失败”不一定是 L0；如果涉及登录、订单、回调、风控或退款，就是 L2。
- “把 OSS 全量切到 CDN”不是简单替换字符串；它涉及资源链路、证书、缓存、回滚和客户端验证，应按 L2。
- “继续这个任务，另外加一个跨对话任务名机制”不是普通继续；新增机制应回到 PRD。
