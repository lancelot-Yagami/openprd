## 新增需求

### 需求：OpenPrd 项目级版本轨道与变化摘要
OpenPrd 现在已经有内部 PRD 版本和用户视角变化摘要，但还没有一个可选的项目级版本轨道来维护当前版本号、版本内更新项，以及与提交标签的协同规则。结果是版本号已经在项目里被真实使用，却缺少一个本地真源来累计本版本的需求、问题修复、交接摘要和最终发布说明。

#### 场景：主流程成功
- **当** 用户在项目里启用版本轨道后，OpenPrd 记录当前版本号，例如 0.1.23，并把后续变化条目累计到该版本下。
- **则** 新增项目级版本账本，支持当前版本、版本状态和版本内变化项存储。

#### 场景：边界情况保持可见
- **当** 项目可能已经有自己的一套版本号规则，OpenPrd 需要允许手动修正当前版本号，而不是强推默认策略。
- **则** 产品应保持该情况明确可见，以支持实现和验证

#### 场景：失败模式得到处理
- **当** 版本轨道和内部 PRD 版本被混用，导致用户误以为 v0003 就是项目发版号。
- **则** 产品应提供有边界且可评审的结果
