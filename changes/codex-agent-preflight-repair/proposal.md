# Codex 代理启动前检查与显式修复

## 背景与原因

当本工具通过代码代理执行循环任务时，如果本机代理命令缺少平台原生组件，当前流程会直接暴露底层错误，任务无法进入子会话，用户也不知道如何修复。

## 变更内容

- Codex 代理循环启动前运行命令健康检查。
- 识别平台原生组件缺失并给出诊断。
- 医生命令支持显式修复 Codex 命令。
- 循环命令提供显式修复快捷入口。
- 更新命令帮助、医生和循环输出、测试和基础后端结构说明。
- 新增 Codex 命令健康检查辅助模块，运行版本检查并返回结构化健康结果。
- Codex 代理循环在真实执行前调用健康检查；失败时不继续启动子会话。
- 识别平台原生组件缺失，输出诊断、缺失组件名和修复命令。
- 医生命令在用户显式请求修复时运行全局安装，并在安装后再次验证。
- 循环命令的修复入口对 Codex 代理复用同一显式修复逻辑。
- 新增检查覆盖版本检查成功、组件缺失、修复成功和修复失败分支。
- 普通输出不再只显示底层运行时错误。
- 默认路径不会静默执行全局安装。

## 能力范围

- `agent-requirements`: Codex 代理启动前检查与显式修复 需求。

## 影响范围

- 主要用户: 使用 openprd loop --run --agent codex 的 OpenPrd 用户
- 主要用户: 维护 OpenPrd agent workflow 的开发者
- 成本来源: 显式修复入口会调用 npm install -g @openai/codex@latest，可能产生网络下载时间和全局环境变更成本。
- 额度限制: 默认 preflight 只运行 codex --version，不做安装。
- 额度限制: 只有 --fix 或 --repair-agent 才允许执行修复命令。
- 额度限制: 测试环境通过模拟 runner 覆盖，不真实联网安装。
- 依赖: 依赖本地 Codex 命令的版本检查行为。
- 依赖: 依赖全局重新安装 Codex 作为官方错误提示给出的修复路径。
- 依赖: 依赖现有 OpenPrd 医生、循环、参数解析、输出和测试结构。
- 风险: 用户全局安装目录权限不足导致修复失败。
- 风险: 自定义代理命令可能不叫 Codex，需要谨慎推导是否检查。
- 风险: 过度包装错误可能隐藏其他真实问题。
