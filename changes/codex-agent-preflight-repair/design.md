# 设计

## 背景

已有真实会话复现了平台原生组件缺失导致循环任务失败。需要在新用户第一次运行代理循环前，提供可理解的健康检查、诊断说明和显式修复入口。

## 目标

- Codex 代理启动前先做命令健康检查。
- 缺少平台原生组件时输出明确诊断和修复命令。
- 只有用户显式请求修复时才执行全局安装。

## 范围

- Codex 代理循环启动前运行命令健康检查。
- 识别平台原生组件缺失并给出诊断。
- 医生命令支持显式修复 Codex 命令。
- 循环命令提供显式修复快捷入口。
- 更新命令帮助、医生和循环输出、测试和基础后端结构说明。

## 约束

- OpenPrd 是基于 Node.js 的命令行工具。
- 循环、医生、参数解析、输出和测试已有独立模块。
- 测试使用本地测试框架，不能依赖真实全局 Codex 或联网安装。
- 不得静默全局安装或升级第三方 CLI。
- 不得暴露凭证或要求用户提供 token。
- 修复入口必须明确说明会运行全局 npm install。
- 依赖本地 Codex 命令的版本检查行为。
- 依赖全局重新安装 Codex 作为官方错误提示给出的修复路径。
- 依赖现有 OpenPrd 医生、循环、参数解析、输出和测试结构。

## 业务护栏

- 成本来源: 显式修复入口会调用 npm install -g @openai/codex@latest，可能产生网络下载时间和全局环境变更成本。
- 额度限制: 默认 preflight 只运行 codex --version，不做安装。
- 额度限制: 只有 --fix 或 --repair-agent 才允许执行修复命令。
- 额度限制: 测试环境通过模拟 runner 覆盖，不真实联网安装。
- 滥用防护: 不把 optional dependency 缺失自动转成静默安装。
- 滥用防护: 修复命令必须在输出中显式展示。
- 滥用防护: 非 Codex agent 和 dry-run 不触发修复。
- 监控信号: doctor/loop JSON 输出包含 preflight ok、diagnostic、repair attempted、exitCode。
- 监控信号: 失败输出保留 stdout/stderr 摘要用于定位权限、网络和 npm 错误。
- 报警阈值: codex --version 失败时 loop 不继续拉起子会话。
- 报警阈值: repair 后 recheck 仍失败时返回失败并提示手动命令。
- 止损动作: 安装失败时停止后续 codex exec。
- 止损动作: 提示用户手动执行 npm install -g @openai/codex@latest 或检查 npm 权限/网络。

## 风险与开放问题

- 假设: Codex 缺平台原生组件时输出中包含可识别的缺失组件名。
- 假设: 全局重新安装是用户可接受的显式修复路径。
- 假设: 医生命令是最自然的健康检查入口。
- 风险: 用户全局安装目录权限不足导致修复失败。
- 风险: 自定义代理命令可能不叫 Codex，需要谨慎推导是否检查。
- 风险: 过度包装错误可能隐藏其他真实问题。
- 问题: 后续是否支持自定义 repair command。
- 问题: 是否将 Codex CLI 健康检查纳入 openprd update 或 setup。
