# 设计

## 背景

用户已明确要求把单独环境从默认倾向收紧成例外机制，只在长程 loop 或明确授权时才建议使用。

## 目标

- 减少默认隔离处理带来的额外复杂度
- 保留长程 loop 和显式授权场景下的隔离建议

## 范围

- run context 推荐条件
- hook 与 CLI 的提示文案
- 相关回归测试

## 约束

- 沿用当前 run context、hook 和 review 机制
- 保持现有 loop 执行方式不变
- 不写入敏感信息
- 依赖当前 OpenPrd run、hook、review 与测试体系

## 业务护栏

- 成本来源: 主要成本来自额外分支、额外上下文和后续合并复杂度。
- 额度限制: 普通续做场景默认保持当前上下文处理，不主动扩展隔离环境。
- 滥用防护: 仍保留用户明确授权和长程 loop 的例外通道。
- 监控信号: 关注 run context 推荐结果、hook 提示文案和回归测试结果。
- 报警阈值: 如果普通续做场景仍出现单独环境建议，就视为未达目标。
- 止损动作: 发现误伤长程 loop 或显式授权场景时，优先回滚推荐条件收紧逻辑。

## 风险与开放问题

- 假设: 用户接受把单独环境建议降为例外机制
- 风险: 历史续做与跨工作区恢复的推荐条件容易互相影响
- 风险: 提示文案和推荐结构如果不同步，用户感知会不一致
- 问题: 后续是否要把这条规则继续升级成硬门禁，另行确认
