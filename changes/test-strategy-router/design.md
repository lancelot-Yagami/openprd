# 设计

## 背景

用户希望 OpenPrd 在测试阶段帮助项目建立分层测试体系，让不同大小和风险的需求获得合适的验证强度，并沉淀到任务、质量报告和 agent skill 中。

## 目标

- 新增测试策略分流器，按风险和证据成本推荐单元测试、集成测试、端到端测试、人工审查、预演、视觉对比或专项验证组合。
- 让 OpenSpec 任务、循环测试报告和质量报告能展示本轮验证计划、执行证据和豁免理由。
- 把测试分流接入生成的 skills、AGENTS 指引和 OpenPrd 质量评估，而不是另起一套平行门禁。

## 范围

- 新增 OpenPrd 测试策略分流指引。
- 扩展 OpenSpec 任务元数据，支持测试层级、测试规模、验证范围、风险或证据计划等验证字段。
- 扩展任务生成，根据需求分流、主流程、非功能、边界和失败路径给出默认测试策略。
- 扩展质量评估执行环境和 HTML 报告，展示单元、集成、端到端、人工、专项验证证据的覆盖情况。
- 扩展循环阶段性测试报告，让单任务执行记录选择的测试策略和证据路径。
- 更新 README、基础文档和相关测试。

## 约束

- OpenPrd 是基于 Node.js 的命令行工具，测试使用 node --test。
- 现有质量评估、循环、OpenSpec 任务、Agent 指引生成和 README 已有验证相关职责，需要在原有链路上扩展。
- 标准 skill 变更会影响 Codex、Claude 和 Cursor 生成物。
- 不得把外部方法论中的比例当成未经项目确认的硬性合规要求。
- 不得让测试证据包含密钥、令牌、个人信息或完整敏感日志。
- 涉及微信小程序验证时仍必须使用 weapp-dev-mcp，不能仅靠普通端到端文字声明。
- 依赖现有 openprd-quality、openprd-requirement-intake、openprd-harness 和 Agent 指引生成机制。
- 依赖 .openprd/quality/evidence、.openprd/harness/test-reports、coverage、tests/reports 等证据源。
- 外部方法论参考 Martin Fowler 测试金字塔、Google 测试规模和端到端测试建议、Kent C. Dodds 测试奖杯以及 Thoughtworks Agent 反馈信号。

## 业务护栏

- 成本来源: E2E、真实浏览器、小程序、visual-compare、性能或极端数据测试会增加本地执行时间和排队成本。
- 成本来源: 默认新增能力不引入第三方服务调用或联网依赖；只有项目已有测试命令要求时才使用对应工具。
- 额度限制: 第一版测试策略以 advisory 和证据提示为主，不把固定比例作为硬性阻断。
- 额度限制: 大型或慢速测试必须有风险理由、执行范围和替代证据说明。
- 额度限制: 缺少本次执行证据时不能宣称已验证。
- 滥用防护: 不允许用旧日志、空测试、脚本存在或截图存在冒充本次测试证据。
- 滥用防护: 不允许把 70/20/10 当成硬指标诱导无意义单测或过度 E2E。
- 滥用防护: 测试豁免必须记录原因和风险。
- 监控信号: quality JSON/HTML 报告展示各测试层级的能力、执行证据、缺口和豁免理由。
- 监控信号: loop test report 记录任务级 test-layer、test-size、test-scope、verify command 和 evidence。
- 监控信号: run --verify 汇总 quality 的 productionReady 与测试证据缺口。
- 报警阈值: 必测层级缺少本次执行证据时显示 needs-evidence。
- 报警阈值: 任务触碰权限、成本、安全、用户主路径、发布或外部依赖却未升级验证时显示需关注。
- 报警阈值: 检测到大量 E2E 但缺少单测或集成/契约测试时提示倒金字塔风险。
- 止损动作: 发现证据不足时要求补测、补证据或记录明确豁免原因。
- 止损动作: 慢测试或 flaky 测试不能反复重跑掩盖，需要记录风险并建议拆出更稳定的中间层验证。
- 止损动作: 涉及小程序运行态时回到 weapp-dev-mcp 实测，不允许普通文字替代。

## 风险与开放问题

- 假设: 多数项目可以通过命令名、目录名、依赖和任务元数据推断测试层级。
- 假设: 用户更需要最小足够证据，而不是固定比例或形式化覆盖率。
- 假设: 新增字段以 advisory 起步，比一开始强阻断更容易兼容历史项目。
- 风险: 过度分流可能增加用户和 Agent 的流程负担。
- 风险: 规则太硬会导致过度 E2E 或无意义单测。
- 风险: 规则太松又可能被 Agent 用旧证据或空测试绕过。
- 风险: 不同技术栈对 integration、component、contract、e2e 的定义不一致。
- 问题: 测试策略字段命名最终使用 test-layer、test-size、test-scope，还是使用 evidence-plan 或 test-strategy。
- 问题: 第一版是否只做提示报告，还是对缺少本次证据的必测层级直接阻断。
- 问题: 是否需要新增 CLI 子命令 openprd test-strategy，或仅作为 skill、质量评估和任务能力存在。
