## 新增需求

### 需求：OpenPrd 测试策略分流器
OpenPrd 现在有质量门禁和验证证据机制，但缺少能按需求风险分流单元测试、集成测试、端到端测试、人工审查和专项验证的统一机制，导致测试要求不够可解释、证据不够结构化。

#### 场景：主流程成功
- **当** 用户提出需求后，OpenPrd 先做需求分流，再由测试策略分流器给出最低验证组合和升级条件。
- **则** OpenPrd 生成的任务支持测试策略元数据，并保留现有完成条件和验证命令行为兼容。

#### 场景：边界情况保持可见
- **当** 小改动触碰权限、数据写入、命令行或接口契约、生成物或用户主路径时，需要从单元测试升级到集成测试或端到端测试。
- **则** 产品应保持该情况明确可见，以支持实现和验证

#### 场景：失败模式得到处理
- **当** Agent 用旧日志、空测试或脚本存在冒充本次证据时，质量评估应提示缺少当前执行证据。
- **则** 产品应提供有边界且可评审的结果
