# OpenPrd 测试策略分流器

## 背景与原因

OpenPrd 现在有质量门禁和验证证据机制，但缺少能按需求风险分流单元测试、集成测试、端到端测试、人工审查和专项验证的统一机制，导致测试要求不够可解释、证据不够结构化。

## 变更内容

- 新增 OpenPrd 测试策略分流指引。
- 扩展 OpenSpec 任务元数据，支持测试层级、测试规模、验证范围、风险或证据计划等验证字段。
- 扩展任务生成，根据需求分流、主流程、非功能、边界和失败路径给出默认测试策略。
- 扩展质量评估执行环境和 HTML 报告，展示单元、集成、端到端、人工、专项验证证据的覆盖情况。
- 扩展循环阶段性测试报告，让单任务执行记录选择的测试策略和证据路径。
- 更新 README、基础文档和相关测试。
- 新增测试策略分流指引，定义风险驱动的证据矩阵和需求分流后的默认验证建议。
- OpenSpec 任务支持测试层级、测试规模、验证范围等测试策略元数据，并在校验中校验取值。
- 任务生成时根据实现、主流程闭环、验收、非功能、边界和失败模式推导默认测试策略。
- 质量评估检测测试命令、测试目录、证据来源和激活任务，输出测试策略矩阵。
- 循环提示和完成报告展示本任务测试策略、执行命令、结果和证据路径。
- Agent 指引、README 和基础文档同步说明测试分流流程。
- OpenPrd 生成的任务支持测试策略元数据，并保留现有完成条件和验证命令行为兼容。
- 新增 openprd-test-strategy skill 或等价的标准指引，并被 Codex、Claude、Cursor 生成物引用。
- 质量评估报告包含测试策略矩阵，不把固定比例作为硬阻断。
- 本地测试覆盖任务元数据解析、任务生成、质量检测、循环报告和生成 skill 更新。

## 能力范围

- `agent-requirements`: OpenPrd 测试策略分流器 需求。

## 影响范围

- 主要用户: 使用 OpenPrd 交付需求并依赖 Agent 自测的开发者
- 主要用户: 维护 OpenPrd quality、loop、tasks 和 agent guidance 的 OpenPrd maintainer
- 成本来源: E2E、真实浏览器、小程序、visual-compare、性能或极端数据测试会增加本地执行时间和排队成本。
- 成本来源: 默认新增能力不引入第三方服务调用或联网依赖；只有项目已有测试命令要求时才使用对应工具。
- 额度限制: 第一版测试策略以 advisory 和证据提示为主，不把固定比例作为硬性阻断。
- 额度限制: 大型或慢速测试必须有风险理由、执行范围和替代证据说明。
- 额度限制: 缺少本次执行证据时不能宣称已验证。
- 依赖: 依赖现有 openprd-quality、openprd-requirement-intake、openprd-harness 和 Agent 指引生成机制。
- 依赖: 依赖 .openprd/quality/evidence、.openprd/harness/test-reports、coverage、tests/reports 等证据源。
- 依赖: 外部方法论参考 Martin Fowler 测试金字塔、Google 测试规模和端到端测试建议、Kent C. Dodds 测试奖杯以及 Thoughtworks Agent 反馈信号。
- 风险: 过度分流可能增加用户和 Agent 的流程负担。
- 风险: 规则太硬会导致过度 E2E 或无意义单测。
- 风险: 规则太松又可能被 Agent 用旧证据或空测试绕过。
- 风险: 不同技术栈对 integration、component、contract、e2e 的定义不一致。
